# Article plan

Fecha de plan inicial: 2026-07-06
Fecha de ajuste editorial: 2026-07-08

## Principio editorial

Los articulos deben explicar problemas reales y mostrar que estoy construyendo soluciones concretas. El codigo aparece solo cuando aclara una decision. No deben sonar como clases introductorias ni como documentacion de librerias.

## Serie 1: Producto y problemas reales

### 1. Nuku: el problema de registrar salud en familia

- Tesis: muchas familias registran presion, glucosa e insulina de forma dispersa. Nuku busca ordenar ese trabajo diario sin convertirlo en una experiencia medica pesada.
- Enfoque:
  1. El problema cotidiano
  2. Por que las apps individuales no cubren bien el cuidado familiar
  3. Que esta construyendo Nuku
  4. Como lo uso como producto de aprendizaje
  5. Como podria servir a otras familias o proyectos
  6. Limites: no diagnostico, no reemplaza criterio medico
- Referencias:
  - American Heart Association: home blood pressure monitoring
  - CDC: diabetes management
  - Next.js, Prisma, Recharts como soporte tecnico

### 2. Dream Home: el problema de decidir una vivienda en Chile

- Tesis: comprar vivienda no es solo calcular un dividendo. La decision mezcla UF, pie, carga financiera, propiedades, mapas, cotizaciones y comparacion.
- Enfoque:
  1. El problema de comparar viviendas
  2. Por que una calculadora sola se queda corta
  3. Que resuelve Dream Home
  4. Como lo uso para convertir una planilla mental en producto
  5. Como otras personas pueden usar este enfoque
  6. Limites: planificacion, no asesoria financiera
- Referencias:
  - CMF Chile: credito hipotecario
  - Banco Central: UF
  - Next.js y herramientas de visualizacion

### 3. AI Memory Vault: el problema de perder la historia de trabajo con agentes

- Tesis: las sesiones con agentes contienen decisiones de ingenieria que se pierden si quedan atrapadas en una herramienta.
- Enfoque:
  1. Que se pierde al cambiar de agente o terminal
  2. Por que la sesion es una pieza de memoria tecnica
  3. Que construye AI Memory Vault
  4. Como lo uso en mi propio flujo
  5. Como otras personas pueden aprovecharlo
  6. Limites: memoria util, no verdad absoluta
- Referencias:
  - SQLite FTS5
  - Claude Code hooks/commands
  - Codex skills/subagents

## Serie 2: Investigacion aplicada

### 4. Vision AI for cultural heritage is not a demo problem

- Tesis: aplicar vision por computador a patrimonio cultural exige pensar en revision experta, datos ambiguos y evidencia visual, no solo en metricas.
- Enfoque:
  1. Que problema investigo
  2. Por que las imagenes patrimoniales son dificiles
  3. Como se puede aprovechar mi investigacion en otros dominios
  4. Que rol deberia tener el modelo
  5. Limites y responsabilidad

### 5. Training YOLO without treating it like magic

- Tesis: entrenar YOLO no es el logro principal. Lo valioso es crear un loop de datos, revision y mejora.
- Enfoque:
  1. Por que falla la mayoria de demos
  2. Dataset antes que modelo
  3. Revision visual antes que tuning
  4. Como transferir el metodo a proyectos reales
